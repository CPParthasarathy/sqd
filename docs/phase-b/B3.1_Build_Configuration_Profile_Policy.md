- Validation date: 2026-07-17

## Purpose

The authoritative ESP-IDF configuration sources are:

| File | Purpose |
|---|---|
| `sdkconfig.defaults` | Common target and hardware invariants |
| `sdkconfig.defaults.debug` | Debug diagnostic overlay |
| `sdkconfig.defaults.validation` | Validation and qualification overlay |
| `sdkconfig.defaults.production` | Production-oriented overlay |

Generated `sdkconfig` files are build outputs and are not authoritative.

## Common platform invariants

```text
CONFIG_IDF_TARGET="esp32s3"
CONFIG_ESPTOOLPY_FLASHSIZE_8MB=y
CONFIG_ESPTOOLPY_FLASHSIZE="8MB"
```

## Profile selection

Apply the common defaults followed by one profile overlay:

```text
SDKCONFIG_DEFAULTS=sdkconfig.defaults;sdkconfig.defaults.<profile>
```

Each profile must use an isolated generated `sdkconfig` and build directory.

## Debug profile

The Debug profile uses debug optimization, enabled assertions, debug logging,
verbose maximum logging, stack-overflow canary checks, light heap poisoning,
print-and-halt panic handling, and UART core dumps.

## Validation profile

The Validation profile uses size optimization while retaining enabled
assertions, informational logging, debug maximum logging, stack and heap
diagnostics, rebooting panic handling, and UART core dumps.

## Production profile

The Production profile uses size optimization, silent assertions, warning-level
logging, disabled dynamic log-level control, stack-overflow canary checks,
disabled heap poisoning, rebooting panic handling, and disabled core dumps.

This profile does not enable secure boot, flash encryption, anti-rollback,
irreversible eFuse settings, or final production partitioning. Those controls
require separate work packages.

## Configuration-control rules

1. Profile changes are reviewed source changes.
2. Generated `sdkconfig` files do not replace controlled defaults.
3. All profiles must resolve under ESP-IDF v6.0.2.
4. All profiles must pass isolated clean builds.
5. Evidence must record configuration, toolchain, binary sizes, and hashes.
6. A profile change invalidates prior build evidence.
7. Builds produced using another ESP-IDF version are nonconforming unless the
   project baseline is formally updated.
